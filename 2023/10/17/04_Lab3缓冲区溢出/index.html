    • Buffer overflow vulnerability and attack

  • 实验环境：SEED Ubuntu 20.04 VM.

三、任务一熟悉：shellcode

1、实验c语言实现的代码：

  • 无法直接使用这个程序的二进制代码作为shellcode，因为可执行文件实际执行时，还要经过操作系统的加载器处理。直接将可执行文件加载到内存中，是无法直接执行的。
  • 最好的办法，还是使用汇编代码。

2、32位的shellcode：

  • eax：必须是11,11是execve的系统调用号
  • ebx：必须保存命令字符串的地址，即”/bin/sh”的地址
  • ecx：必须保存参数数组的地址，如char* name[2]中的name，第一个元素name[0]=”/bin/sh”，第二个参数name[1]=NULL，即等于0，表示数组的结尾。
  • edx：必须保存传给新进程的环境变量的地址，可以设置为NULL即0，表示不传递环境变量。

3、64位的shellcode：

  • 与32位的比较：
    • rdx与32位edx功能相同，都是环境变量，可以是0。
    • 指令/bin//sh的传入，直接存在rax，再将rax入栈！
    • rdi与32位的ebx相同，保存/bin//sh的地址
    • rsi与32位的ecx相同，保存name数组的起始地址
    • rax与32的eax相同，保存系统调用号，但是是59。
    • 执行系统调用使用syscall而非int 0x80

4、调用shellcode：

  • 此任务是在程序中直接调用shellcode，体会shellcode的使用，而非构造缓冲区溢出漏洞的shellcode。

  • 需要将前面的汇编代码转换为二进制形式才能用于组成shellcode，具体如何转换参考Shellcode实验。

  • call_shellcode.c：

    • #if else条件编译。
    • shellcode前面都插入了一段setuid(0)的机器指令，将进程RUID设置为0。
    • int (func)() = (int()())code; 函数指针的声明和初始化：
      • int (func)()：这部分是函数指针的声明。它表明 func 是一个指向函数的指针，该函数没有参数，并且返回一个整数（int）。*函数指针的声明采用 (returnType (*pointerName)(parameterTypes)) 的形式。
      • = (int(*)())code;：将code强制转换称为函数指针类型(int(*)())。即将 func 初始化为指向 code 的地址。
    • func();调用函数指针func指向的函数，即执行code中的机器代码。
  • 使用make编译这段程序，Makefile如下：
    • all目标：默认make指令下执行
    • setuid目标：make setuid指令下执行，编译的同时，生成root的setuid程序
    • clean目标：make clean指令执行，清理文件
  • 首先是默认make指令生成seed权限的可执行文件，执行，发现都获得了seed权限的shell：

  • make setuid指令，生成root权限的Set-UID程序，执行获得root权限的shell

五、任务三：使用32位程序攻击level1

1、调查：

  • 使用make stack-L1指令，生成32位的可执行文件stack-L1，和带有调试信息的stack-L1-dbg

  • gdb调试：
    • b bog，next
    • p $ebp，0xffffcae8
    • p &buffer，0xffffca7c
    • ebp-buffer=108

  • ret在badfile中的相对位置：ebp+4=buffer+108+4=buffer+112，即ret从badfile中的112字节开始。
  • ret的值：需要大于ebp+4，即0xffffcaf8+4，因为gdb中含有调试的信息，需要偏大一点，修正为0xffffcaf8+100

2、构造shellcode：

  • 完善exploit.py：
    • shellcode在badfile中的起始位置为，517-len(shellcode)
    • 返回地址ret的值，大于ebp+4，修正为ebp+100即0xffffcaf8+100
    • ret字段在badfile上的起始字节offset=ebp+4=buffer+112，即112字节
  • 运行exploit.py，得到badfile。

3、实施攻击：

  • 执行./stack-L1，出现段错误

  • 多次尝试修正exploit.py文件，但结果都是Segmentation fault，怀疑不是步骤的问题，而是运行的shell的问题，在调试stack-L1-dbg程序时，一句一句执行机器指令，能正常触发/bin/sh，在输入一条指令后，能成功执行看到结果，但是会导致shell崩溃。所以合理怀疑stack-L1运行时，可能也正确的执行了漏洞，触发了/bin/sh，但是不知道什么原因，程序崩溃了。

  • 选择直接调试stack-L1，观察能否触发/bin/sh：

    • 首先在bof设置断点，运行到该断点后，一句一句执行，观察bof函数运行结束时，能否正常跳转执行恶意插入的栈指令：

    • 发现在bof函数ret之后，gdb显示执行的代码已经跳转到了栈中，而其中的指令正好是插入nop指令，说明我们构造的badfile是正确的，已经成功利用了缓冲区溢出漏洞！
    • 继续适当的next，定位到执行shellcode，观察执行到shellcode的最后一条指令int 80会发生什么：

    • 发现正常触发了/bin/sh，但是只能执行一条指令，且执行后依然崩溃！

六、任务四：不知缓冲区大小下的攻击level2

  • 实验要求不能通过gdb获得buffer地址，只能获得ebp，从而无法计算出ebp-buffer，即无法推断出ret字段，在badfile文件中的位置，且不能暴力的遍历所有可能的ret字段，一次攻击只能尝试一次可能的ret字段位置。
  • 直接修改exploit.py的offset的值即可，因为buffer的大小为100到200字节，所以依次选取offset的值为100,104,108,112,……,200
  • 首先，调试stack-L2-dbg，获得ebp的值，从而设置ret的值为ebp+100。

  • 然后不断修改exploit.py，生成不同的ret字段在不同位置的badfile，直到触发/bin/sh。
  • 在ret字段在错误位置时，如offset=112，此时程序实际接收到的跳转地址为4个NOP指令，会跳转执行0x90909090地址的指令，gdb调试如下：

  • 多次尝试之后，发现offset=172时，能成功跳转执行恶意代码

七、任务五：攻击64位程序level3

1、调查：

  • 使用gdb调试stack-L3-dbg，获取地址信息：
    • rbp=0x7fffffffd930
    • rbp-buffer=208

2、构造shellcode：

  • 根据获取的rbp值，ret字段的值应该设置为rbp+8=0x7fffffffd930+8，考虑到调试信息，修正为0x7fffffffd930+80（应该确保为8的倍数）。
  • ret字段在badfile中的位置，即offset=208+8=216，L=8
  • 对应的shellcode使用64位的shellcode（参考03shellcode实验）
  • 得到如下exploit.py：

3、实施攻击：

  • 运行exploit.py和stack-L3，出现Illegal instruction，执行的机器指令错误
    • 可能是rbp错误导致ret不准确（排除）
    • 可能是ret的偏移80太小了，导致ret没有跳转到恶意指令。

  • 修正80改为160，再次运行，发现报的错是Segmentation fault，考虑到前面的原因，此时的ret正常跳转执行了恶意指令。

  • 使用gdb调试运行stack-L3：

    • 定位到bof即将结束，执行ret前：

    • next，发现gdb显示执行的指令地址已经变成了栈区地址，说明漏洞触发成功，只是原本应该是NOP的指令被识别成了ADD指令？？

    • 想要再适当的next，找到执行shellcode时，发现程序会被阻塞，gdb显示程序Program received signal SIGSEGV, Segmentation fault.，Stopped reason: SIGSEGV。（此处应该就是直接运行stack-L3出现段错误的原因了）

    • 无论怎么next，代码区执行的指令地址都不会变了。

  • 检索了一下Program received signal SIGSEGV, Segmentation fault，出现这种错误的原因有两种：

    • 第一种：程序试图访问一个未分配给它的内存区域。排除这种情况，因为因为根据源程序，dummy_function函数在调用bof函数之前，申请了大概1000字节的局部变量。而我们要执行的机器指令，最多也只能超过bof函数帧517字节，不可能超过main函数栈帧，更不能访问到不属于这个程序的内存空间了。
    • 第二种：尝试执行不允许的操作，如读写只读内存区域。
      • 详细查了一下add BYTE PTR [rax],al指令，发现这是编译器对大量NOP操作的优化，只是简单的字节加法操作，不会导致段错误。（变相证明了成功跳转到了badfile，因为只有大量NOP才会优化出大量add BYTE……指令）
      • 还有一种可能是不可执行栈，但是编译stack-L3时已经加了相应选项……

  • 后面突然意识到，badfile中0的问题。返回地址的值ret=0x7fffffffd930+160，只使用了6个字节，意味着还有两个字节的0，这可能又会导致strcpy提前中断。（疑问：如果提前中断，为什么跳转后的指令出现大量NOP呢？？理论上跳转的地址是比ret字段更后面的，如果中断，这里不应该有NOP指令）

  • 时间不够了，不想再研究，希望以后的同学能发现点什么

八、任务六：不知缓冲区大小下攻击level4

  • 类似于level，不过buffer非常小，为10。

  • 未知的buffer主要影响的是ret字段在badfile中的位置

  • gdb调试stack-L4-dbg

  • 出现的问题：缓冲区长度不是8的倍数时，如果每次猜测ret字段的位置仍然是加8字节形式，可能导致永远无法猜测出缓冲区大小！
  • 如实际缓冲区为10时：正确的ret字段应该从18字节开始
    • 第一次ret从0字节开始，第二次ret从8字节开始，第三次16，第四次24，显然不能猜测到正确的位置18。
  • 解决方法，每次只加1字节，即offset从1开始，1,2,3……，
  • offset=1等不正确的位置时，运行stack-L4程序会崩溃

  • offset=18时，正确的位置，成功获得shell：

1、攻破方法：

  • 只需要在shellcode前面加上setuid(0)的系统调用，将RUID设置为0，就可以攻破这种机制：
  • 测试是否有效，使用shellcode文件夹，执行make setuid，会自动编译，并设置可执行文件未Set-UID程序

十一、任务九：尝试其他防御策略

1、启动StackGuard：

  • 许多编译器实现了栈溢出检测机制
  • gcc -fno-stack-protector可以关闭栈溢出检测，尝试不关闭的效果
  • 测试stack-L3不关闭StackGuard的效果，发现检测出栈溢出，直接终止程序

2、启动不可执行栈：

  • 编译时不使用-z noexecstack
  • 测试stack-L3启动不可执行栈，发现直接检测出段错误
